## What was done

- Shipped **P4 / S3 Flashcards** (final-sprint stretch): a `/flashcards` Telegram command that generates flashcards from the user's uploaded materials and sends them to the chat, one card per message.
- Added `backend/app/services/flashcard.py` — `generate_flashcards()` reuses the proven quiz-generation pipeline: RAG-retrieve the user's material chunks → ask the LLM for a JSON array of `{front, back, concept}` (`kind="flashcard_generate"`, so the call is logged through the existing monitoring path) → return the cards. No persistence — cards are ephemeral.
- Exposed a shared-secret `POST /telegram/flashcards/generate` endpoint mirroring `bot_quiz_generate`, plus the bot client method and `/flashcards` command handler; updated the `/start` help copy.
- Added parser unit tests (`test_flashcard_generation.py`) covering raw and markdown-fenced JSON arrays.
- Merged via PR #24 (4 small Conventional commits), branch deleted, `TODO.md` updated.

## Problems encountered

- No real blockers. The main open question was scope: the spec said "generate from a document; send to Telegram," which could mean anything from a Telegram-only MVP to full DB persistence + a web UI.
- Couldn't run the live `/flashcards` end-to-end locally — that needs a running bot + backend + a linked Telegram chat + uploaded materials, none of which are wired up on this machine.

## Solutions

- Confirmed scope with the user before building: **Telegram-only, ephemeral** — no DB table, no Alembic migration, no web UI. That kept the diff thin and risk low by leaning entirely on the existing `/quiz` flow.
- Verified everything that's checkable offline instead: `ruff` clean, `pytest` green (incl. new parser tests), endpoint registered, and service/bot files import/parse cleanly. Left the live Telegram run flagged as the recordable acceptance flow for the demo.
